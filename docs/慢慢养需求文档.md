## 2. 产品目标

慢慢养是一款面向院外长期营养管理场景的 App，核心目标是结合医院营养处方或 AI 生成营养方案，帮助用户完成日常营养记录、风险提示、动态调整和周期性反馈。

摄像头、视觉识别和视觉算法相关能力由算法开发文档定义，本需求文档仅保留产品侧占位和依赖关系。

产品同时结合“爱人如养花”的概念，将 7 天营养管理周期转化为花园成长反馈，为长期慢性病用户提供心理关怀和持续管理动力。

## 3. Demo 范围

### 3.1 Demo 包含

1. 用户进入 App 后选择病症。
2. 用户填写基础身体指征。
3. 用户可导入医院营养处方。
4. 用户无营养处方时，可根据身体指征生成营养方案。
5. App 展示每日总目标、每餐建议、食谱、营养素目标、禁忌食物、补充剂建议。
6. App 按餐拆分营养指标。
7. 摄像头、视觉识别和视觉算法相关能力由算法开发文档定义，本需求文档不展开。
8. 根据当前用户营养方案判断食物符合程度，并生成不同颜色描边的食物贴纸。
10. 红色风险食物触发弹窗提示。
11. 记录用户真实饮食量，并根据摄入情况动态调整后续推荐。
12. 以 7 天为固定周期展示花园成长反馈。
13. 在记录界面展示日、周、月维度的摄入图表。
14. Agent 主动通过文字询问用户食谱反馈和身体感受。
15. Agent 支持调整食谱和院外陪伴。
15. 摄像头与视觉相关隐私授权、说明和管理。

### 3.2 Demo 不包含

1. 营养师端或后台。
2. 家属端。
3. 独立通知提醒功能。
4. 更复杂的花园成长体系，例如断签、补签、等级、奖励等。

## 4. 用户与使用场景

### 4.1 目标用户

Demo 阶段面向存在院外营养管理需求的人群，尤其包括长期慢性病用户、术后康复用户和肿瘤康复用户。

### 4.2 Demo 病症类型

用户进入 App 后需要先选择病症。Demo 阶段病症类型固定为：

1. 糖尿病
2. 高血压
3. 内科术后康复
4. 肿瘤康复
5. 其他

当用户选择“其他”时，用户可以自行填写病症名称。

### 4.3 核心痛点

1. 用户不知道食物或补充产品的克数。
2. 用户难以长期跟踪院外营养摄入情况。
3. 图片和文字记录无法准确反映具体进食量。
4. 用户需要 AI 主动反馈，并根据摄入情况调整营养结构。
5. 部分用户记性不好，记不住是否吃过、吃了什么。
6. 用户需要监督和反馈。
7. 用户需要连续性视频跟踪饮食过程。
8. 现阶段院外营养管理多聚焦短周期，长期管理仍需要调控能力。

## 6. 用户主流程

### 6.1 首次进入流程

1. 用户进入 App。
2. 用户选择病症。
3. 用户填写基础身体指征。
4. App 根据用户是否已有医院营养处方进入不同路径。

### 6.2 有医院营养处方路径

1. 用户选择导入医院营养处方。
2. 用户上传处方文件。
3. App 读取处方内容。
4. App 生成或展示对应营养方案。
5. App 按餐拆分营养指标。

支持导入格式：

1. PDF
2. Word，包括 `.doc` 和 `.docx`

### 6.3 无医院营养处方路径

1. 用户确认无医院营养处方。
2. App 根据用户基础身体指征生成营养方案。
3. App 展示营养方案。
4. App 按餐拆分营养指标。

生成营养方案使用的用户信息包括：

1. 体重
2. 身高
3. 年龄
4. 性别
5. 过敏史
6. 疾病史
7. 工作强度

## 7. 基础身体指征

### 7.1 字段要求

| 字段 | 说明 |
| --- | --- |
| 体重 | 用户填写 |
| 身高 | 用户填写 |
| 年龄 | 用户填写 |
| 性别 | 用户选择或填写 |
| 过敏史 | 用户填写 |
| 疾病史 | 用户填写 |
| 工作强度 | 用户选择 |

### 7.2 字段形式

字段使用国际单位。

工作强度选项固定为：

1. 低
2. 中
3. 高

字段的具体输入控件和校验规则未在当前需求中定义，需在后续交互或开发设计阶段补充。

## 8. 营养方案

### 8.1 方案来源

营养方案来源分为两类：

1. 医院营养处方导入。
2. 根据用户基础身体指征 AI 生成。

### 8.2 方案展示内容

信息界面需要展示以下内容：

1. 每日总目标
2. 每餐建议
3. 食谱
4. 营养素目标
5. 禁忌食物
6. 补充剂建议

### 8.3 营养指标拆分

营养指标按餐拆分。

需要支持的营养指标包括：

1. 能量，单位 kcal
2. 蛋白质，单位 g
3. 脂肪，单位 g
4. 碳水化合物，单位 g
5. 膳食纤维
6. 钙
7. 镁
8. 维生素

未明确单位的指标包括：

1. 膳食纤维
2. 钙
3. 镁
4. 维生素

这些指标单位和计算规则先按照国际认可度最高的标准执行。

## 9. 信息界面

### 9.1 页面定位

信息界面用于承载营养方案、按餐营养目标、摄入识别结果、食物贴纸和风险反馈。

### 9.2 页面内容

信息界面需要包含：

1. 营养方案展示区域。
2. 按餐拆分的营养指标。
3. 食物描边贴纸展示区域。
4. 贴纸颜色图例说明。
5. 摄入记录相关信息。
6. 风险提示和动态调整说明。

### 9.3 食物贴纸

手机摄像头捕捉食物并拍照后，AI 抠图将实际食物变为食物贴纸。

贴纸根据当前用户营养方案判断符合程度，并生成不同颜色描边：

| 判断结果 | 描边颜色 | 色值 |
| --- | --- | --- |
| 符合营养进食 | 草绿色 | `#9DCF55` |
| 一般符合 | 黄色 | `#EFD67C` |
| 非常不符合 | 红色 | `#C82727` |

### 9.4 图例说明

信息界面需要在明显位置展示贴纸颜色图例说明。

图例需说明三种颜色对应含义：

1. 草绿色：符合营养进食。
2. 黄色：一般符合。
3. 红色：非常不符合。

### 9.5 贴纸存储位置

食物描边贴纸存储并展示在 App 信息界面最上端。

## 10. 摄像头与视觉识别

### 10.1 文档边界

摄像头、视觉识别和视觉算法相关内容在本需求文档中留白，由算法开发文档详细定义。

本需求文档仅保留产品侧需要依赖的识别结果。

### 10.2 产品侧依赖结果

产品侧需要算法模块提供以下结果：

1. 所有进食行为。
2. 营养补充行为。
3. 特医产品。
4. 特膳产品。
5. 品类名称。
6. 饮食或补充产品克数。
7. 时间。
8. 可用于判断贴纸颜色和营养摄入的数据。

### 10.3 不在本文档展开的内容

以下内容不在本文档展开：

1. 视觉算法如何估算克数。
2. 视觉算法准确率要求。
3. 识别失败、遮挡、光线不足、多人进食等异常场景处理。
4. 食物品类库和特医、特膳产品库。
5. 摄像头捕捉流程。
6. 视频或图像处理流程。

## 11. 风险提示

### 11.1 触发条件

当系统识别到用户准备进食红色风险食物时，触发风险弹窗。

红色风险食物指根据当前用户营养方案判断为“非常不符合”的食物。

### 11.2 弹窗内容

风险弹窗需要提供：

1. 替代食物。
2. 原因说明。
3. 继续进食确认。
4. 通知家属或营养师。

### 11.3 Demo 边界

当前 Demo 不包含家属端和营养师端。

因此，“通知家属或营养师”在 Demo 中按紧急联系人短信通知方式实现：

1. 用户需要输入紧急联系人。
2. App 调用手机短信权限。
3. Demo 阶段必须先弹窗请求用户确认。
4. 用户确认后才允许进入短信发送流程。
5. App 不能直接自动执行短信发送。

## 12. 饮食量记录与动态调整

### 12.1 当餐记录

系统需要根据真实饮食量记录当餐元素指标数据。

### 12.2 超量提醒

当饮食量超出限定范围时，系统提醒用户。

限定范围来源包括：

1. 医院营养处方。
2. AI 生成方案。

### 12.3 下一餐调整

如果单餐饮食超过推荐范围，系统自动调整下一餐推荐饮食量和饮食种类。

系统调整后需要展示调整原因。

### 12.4 剩余可补充量调整

如果用户没有根据食谱进食，系统需要结合用户的进食指标参数，调整剩余可补充量。

调整范围包括：

1. 当天剩余摄入量。
2. 当前周期剩余摄入量。

## 13. 花园界面

### 13.1 页面定位

花园界面用于承载“爱人如养花”的心理关怀概念，并展示 7 天营养管理周期内的完成反馈。

### 13.2 周期规则

Demo 阶段以 7 天为固定周期。

### 13.3 成长规则

1. 用户完成一天的营养管理，且符合管理要求，即可种下一朵小花。
2. 用户连续 7 天完成营养管理，可养成一朵大花。

### 13.4 当日完成判定

一天营养管理完成并符合要求，需要同时满足：

1. 摄入量达标。
2. 记录完整。
3. 无红色风险食物。

### 13.5 Demo 不包含

Demo 阶段不考虑更多花园功能，例如：

1. 断签。
2. 补签。
3. 花园等级。
4. 奖励体系。
5. 更多成长状态。

## 14. 记录界面

### 14.1 页面定位

记录界面用于展示用户摄入情况、食谱反馈和 Agent 文字交互。

### 14.2 摄入情况

记录界面需要展示一定周期内各类元素摄入情况。

周期维度支持：

1. 日
2. 周
3. 月

展示形式为图表。

图表类型规则：

1. 占比展示使用饼状图。
2. 对比展示使用柱状图。

### 14.3 食谱反馈

Agent 通过文字主动询问用户：

1. 吃得怎么样。
2. 身体感觉怎么样。

### 14.4 Agent 对话

Agent 支持：

1. 调整食谱。
2. 院外陪伴。

Demo 阶段 Agent 不扩展到其他能力。

## 15. 隐私与权限

### 15.1 需要涉及的权限

由于产品涉及摄像头、视觉识别、文件上传和短信通知，Demo 需要包含隐私与权限相关需求。

至少涉及：

1. 摄像头权限。
2. 文件上传权限或文件选择能力。
3. 手机短信权限。
4. 可能涉及的本地或云端数据临时存储授权。

### 15.2 摄像头授权说明

用户启用摄像头识别前，需要展示明确说明：

1. 摄像头用于识别进食和特医、特膳产品摄入行为。
2. 摄像头用于估算食物或产品克数。
3. 摄像头可能持续捕捉用户进食全流程。
4. 识别结果会用于营养记录、风险提示和方案动态调整。
5. 摄像头、视觉识别和视觉算法相关细节由算法开发文档定义。

### 15.3 数据管理

摄像头与视觉识别产生的数据仅在识别过程中到报告生成之前临时保存。

数据删除规则：

1. 用户确认生成报告后，相关临时数据自动删除。
2. 用户未确认报告且超过 15 分钟，相关临时数据自动删除。
3. 系统不长期保存摄像头识别记录。

以下内容由摄像头与视觉算法文档定义，本需求文档不展开：

1. 是否允许用户关闭连续性视频跟踪。
2. 原始视频、原始图片和识别结果在临时保存阶段的具体处理方式。
3. 本地处理或云端处理方式。
4. 视觉识别过程中的异常数据处理。

## 16. 文件导入

### 16.1 支持格式

医院营养处方导入支持：

1. PDF
2. Word，包括 `.doc` 和 `.docx`

PDF 支持扫描件识别。

### 16.2 导入结果

导入后，系统需要基于处方内容形成营养方案，并在信息界面展示。

导入后交互要求：

1. 用户可以预览导入和识别结果。
2. 用户不可直接修改识别结果。
3. 如果识别结果不符合预期，用户只能删除后重新上传。

### 16.3 未定义内容

以下内容尚未定义：

1. 导入失败时的提示。
2. 扫描件识别失败时的处理方式。
3. 删除后重新上传的次数限制。

## 17. 算法与规则待定义项

以下内容已确认属于产品能力，但具体规则需要由算法或营养规则模块进一步定义：

1. 视觉算法如何估算食物和特医、特膳产品克数。
2. 视觉算法准确率要求。
3. 识别失败、遮挡、光线不足、多人进食等异常场景处理。
4. 食物品类库和特医、特膳产品库。
5. 摄像头和视觉相关流程。
6. 根据当前用户营养方案判断食物符合程度的规则。
7. 草绿色、黄色、红色三档判断阈值。
8. 替代食物推荐规则。
9. 超出限定范围的判断规则。
10. 下一餐饮食量和饮食种类调整规则。
11. 当天剩余摄入量调整规则。
12. 当前周期剩余摄入量调整规则。
13. 各营养指标的推荐范围和计算方式。

## 18. 非功能需求

### 18.1 适老化要求

产品定位为适老化院外营养管理平台，因此界面设计和交互需要考虑适老化使用体验。

适老化界面按照标准 App 设计要求执行，不设置语音输入。

具体字号、对比度、按钮尺寸和误触处理规则可在 UI 设计阶段细化。

### 18.2 可解释性

系统在以下场景中需要向用户展示原因：

1. 红色风险食物提示。
2. 替代食物推荐。
3. 下一餐推荐饮食量和饮食种类调整。
4. 剩余可补充量调整。

### 18.3 可追踪性

系统需要支持按时间和餐次追踪用户摄入情况，以支撑日、周、月图表展示。
